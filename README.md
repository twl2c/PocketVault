# PocketVault

**PocketVault** — это автономный аппаратный менеджер паролей на базе платы **M5Stack (ESP32-S3, вариант StickS3)**. Устройство хранит учётные записи (название, логин, пароль) непосредственно в энергонезависимой памяти микроконтроллера, шифрует чувствительные данные алгоритмом AES-256 и защищает доступ мастер-паролем. Управление осуществляется двумя физическими кнопками, а данные можно дополнительно читать/писать с компьютера через последовательный (Serial/USB) интерфейс.

Проект решает задачу хранения паролей "офлайн", без облака и без привязки к смартфону — все данные остаются на самом устройстве.

---

# Возможности

- **Защита мастер-паролем.** При первом включении устройство предлагает задать мастер-пароль (с обязательным подтверждением). При последующих включениях требуется ввод пароля для разблокировки.
- **Хранение учётных записей.** Вместимость хранилища — до 10 аккаунтов (`MAX_ACCOUNTS = 10`). Для каждого аккаунта хранятся название, логин и пароль.
- **Шифрование данных.** Логин и пароль каждого аккаунта шифруются алгоритмом **AES-256-CBC** перед записью в память; ключ шифрования выводится из мастер-пароля через SHA-256 с отдельным контекстом (не совпадает с хэшем, используемым для проверки пароля).
- **Просмотр аккаунта.** Пароль по умолчанию скрыт (`********`), кнопкой можно показать/скрыть значение.
- **Добавление аккаунта.** Экранная клавиатура посимвольного ввода (перебор символов кнопками, выбор — подтверждение) для полей Name / Login / Password.
- **Редактирование аккаунта.** Отдельный экран, предзаполняющий текущие значения полей и позволяющий их изменить.
- **Навигация по меню.** Главное меню (Vault / Settings / About), меню хранилища с прокруткой (видно 4 строки, автопрокрутка при выходе выбранного пункта за пределы окна).
- **Управление кнопками.** Поддержка короткого и долгого нажатия (порог удержания — 800 мс) для каждой из двух кнопок, что даёт 4 логических события: SELECT, NEXT, PREV, BACK.
- **Сохранение данных.** Явное сохранение в энергонезависимую память (NVS/Preferences) после добавления/редактирования аккаунта или по команде `SAVE` через Serial.
- **Связь с ПК (Serial API).** Командный протокол через USB-CDC: получение списка аккаунтов, получение одного аккаунта, добавление, удаление, принудительное сохранение.
- **Экран Settings** — присутствует в навигации, но экранная логика не реализована (только заголовок).
- **Экран About** — присутствует в навигации, но экранная логика не реализована (только заголовок).
- **Экспорт/резервное копирование, GUI-приложение для ПК, генератор паролей** — не реализовано (см. раздел «Future Development»).

# Hardware

Точная модель платы не указана явно в коде, но из `platformio.ini` можно определить конфигурацию сборки:

| Параметр | Значение |
|---|---|
| Плата (`board`) | `esp32-s3-devkitc-1` |
| Окружение (`env`) | `m5stack-sticks3` (указывает на устройство семейства **M5Stack StickS3**) |
| Платформа | `espressif32@6.12.0` |
| Фреймворк | `arduino` |
| Микроконтроллер | ESP32-S3 |
| Память | PSRAM включена (`BOARD_HAS_PSRAM`), режим `qio_opi` |
| Разметка флеша | `default_8MB.csv` (плата с 8 МБ флеш-памяти) |
| USB | USB CDC при загрузке включён (`ARDUINO_USB_CDC_ON_BOOT=1`), режим USB — `ARDUINO_USB_MODE=1`; MSC и DFU отключены |
| Дисплей | Встроенный экран, управляемый через библиотеку **M5Unified** (`M5.Display`) |
| Кнопки | Две физические кнопки — `M5.BtnA` и `M5.BtnB` |
| Питание | Не описано в коде; для плат серии M5Stick — встроенный аккумулятор + питание/зарядка по USB-C |
| Интерфейсы | USB (Serial/CDC) для связи с ПК |

Библиотечная зависимость: **M5Unified** (`https://github.com/m5stack/M5Unified`) — единая библиотека абстракции дисплея, кнопок и питания для устройств M5Stack.

# Software Architecture

Проект построен по слоистой архитектуре с чётким разделением ответственности: точка входа инициализирует единый объект `App`, который владеет всеми подсистемами и экранами.

```
main.cpp
  |
  App                     (инициализация и главный цикл)
  |
  +-- Display              (низкоуровневая отрисовка на экран)
  +-- Buttons               (опрос кнопок, события SELECT/NEXT/PREV/BACK)
  +-- Navigation             (переключение текущего экрана)
  |     |
  |     +-- Screen (интерфейс)
  |           |
  |           +-- MasterPasswordScreen
  |           +-- MainMenuScreen
  |           +-- VaultScreen
  |           +-- AccountScreen
  |           +-- AddAccountScreen
  |           +-- EditAccountScreen
  |           +-- SettingsScreen
  |           +-- AboutScreen
  |
  +-- Vault                 (аккаунты в оперативной памяти)
  +-- Storage                (шифрование + энергонезависимое хранение)
  |     |
  |     +-- AesCipher         (AES-256-CBC)
  |     +-- PasswordUtil       (SHA-256, деривация ключа)
  |
  +-- SerialManager          (командный протокол по USB)
```

**Назначение модулей:**

- **App** — точка сборки приложения. Создаёт все объекты, связывает их указателями друг с другом (dependency injection вручную через `init()`) и на каждой итерации `loop()` опрашивает кнопки и обновляет `Navigation` и `SerialManager`.
- **Navigation** — простой менеджер экранов: хранит указатель на текущий и один предыдущий экран, вызывает `draw()`/`update()` активного экрана.
- **Screens** — каждый экран реализует интерфейс `Screen` (`draw()`, `update(event)`) и содержит свою логику отображения и обработки ввода.
- **Vault** — модель данных в оперативной памяти: массив аккаунтов фиксированного размера и операции над ним (добавление, удаление, обновление, поиск).
- **Storage** — сохраняет/загружает данные `Vault` в энергонезависимую память ESP32 (namespace `Preferences`), прозрачно шифруя логин и пароль через `AesCipher`.
- **SerialManager** — реализует текстовый протокол поверх USB Serial для управления хранилищем с компьютера.

# Main Components

**App**
- Назначение: точка сборки и жизненного цикла приложения.
- Основные методы: `init()`, `update()`.
- Связи: владеет всеми экранами, `Display`, `Buttons`, `Vault`, `Navigation`, `Storage`, `SerialManager`; в `init()` связывает их между собой указателями.

**Navigation**
- Назначение: переключение между экранами, возврат на один экран назад.
- Основные методы: `init()`, `setScreen()`, `update()`, `draw()`, `back()`.
- Связи: работает с любым объектом, реализующим `Screen`. Хранит только один уровень истории (`previousScreen`), а не полноценный стек.

**Screen (интерфейс)**
- Назначение: контракт для всех экранов приложения.
- Основные методы: чисто виртуальные `draw()`, `update(ButtonEvent)`.
- Связи: реализуется всеми конкретными экранами.

**MasterPasswordScreen**
- Назначение: установка мастер-пароля при первом запуске и разблокировка при последующих.
- Основные методы: `init()`, `confirm()`, `appendChar()`, `backspace()`, `draw()`, `update()`.
- Связи: использует `Storage` (хэш пароля, ключ шифрования), `Vault` (инициализация демо-данных при первой настройке), `PasswordUtil`; после успеха переключает `Navigation` на `MainMenuScreen`.

**MainMenuScreen**
- Назначение: главное меню (Vault / Settings / About).
- Основные методы: `init()`, `draw()`, `update()`, `getSelected()`.
- Связи: использует `Menu` (модель списка), переключает `Navigation` на `VaultScreen`, `SettingsScreen` или `AboutScreen`.

**VaultScreen**
- Назначение: список сохранённых аккаунтов с прокруткой и пунктом добавления нового.
- Основные методы: `init()`, `draw()`, `update()`, `updateScroll()`.
- Связи: использует `VaultMenu` и `Vault`; открывает `AccountScreen` или `AddAccountScreen`.

**AccountScreen**
- Назначение: просмотр одного аккаунта, показ/скрытие пароля.
- Основные методы: `init()`, `openAccount()`, `draw()`, `update()`.
- Связи: читает данные из `Vault`; долгое нажатие (`PREV`) открывает `EditAccountScreen`.

**AddAccountScreen / EditAccountScreen**
- Назначение: посимвольный ввод и редактирование полей Name/Login/Password через экранную "клавиатуру" (перебор символов из `CHARSET`).
- Основные методы: `appendChar()`, `backspace()`, `nextField()`, `save()`, `draw()`, `update()`.
- Связи: пишут изменения в `Vault` и вызывают `Storage::save()`.

**SettingsScreen / AboutScreen**
- Назначение: заглушки экранов, доступные из главного меню.
- Основные методы: `draw()` (выводит только заголовок), `update()` (обрабатывает только `BACK`).
- Связи: функциональность внутри экранов **не реализована**.

**Vault**
- Назначение: хранит аккаунты в оперативной памяти, управляет добавлением/удалением/обновлением, предоставляет данные экранам.
- Основные методы: `init()`, `getCount()`, `get()`, `add()`, `remove()`, `update()`, `clear()`, `exists()`.
- Связи: используется практически всеми экранами и `SerialManager`; данные персистентно сохраняются через `Storage`.

**Storage**
- Назначение: сохранение и загрузка аккаунтов и мастер-пароля в энергонезависимую память, прозрачное шифрование чувствительных полей.
- Основные методы: `init()`, `save()`, `load()`, `hasMasterPassword()`, `saveMasterPasswordHash()`, `loadMasterPasswordHash()`, `setEncryptionKey()`.
- Связи: использует `AesCipher` для шифрования логина/пароля, `Preferences` (Arduino ESP32 API) как физическое хранилище.

**AesCipher**
- Назначение: симметричное шифрование/дешифрование строк.
- Основные методы: `encryptToHex()`, `decryptFromHex()`.
- Связи: используется `Storage` для шифрования полей `login`/`password`.

**PasswordUtil**
- Назначение: хэширование мастер-пароля и деривация ключа шифрования.
- Основные методы: `sha256Hex()`, `sha256Bytes()`, `deriveKey()`.
- Связи: используется `MasterPasswordScreen` для проверки пароля и получения ключа для `Storage`.

**SerialManager**
- Назначение: обработка команд, поступающих по USB Serial, для управления `Vault` и `Storage` с компьютера.
- Основные методы: `init()`, `update()`, `handleCommand()`, `handleAddInput()`, `cmdList()`, `cmdGet()`, `cmdAdd()`, `cmdDelete()`, `cmdSave()`.
- Связи: напрямую обращается к `Vault` (чтение/изменение) и `Storage` (сохранение).

**Display**
- Назначение: низкоуровневая отрисовка всех экранов через `M5.Display`.
- Основные методы: `init()`, `drawMenu()`, `showTitle()`, `drawAccount()`, `drawAddAccount()`, `drawPasswordPrompt()`.
- Связи: вызывается всеми экранами для вывода информации.

**Buttons**
- Назначение: опрос физических кнопок `M5.BtnA`/`M5.BtnB`, различение короткого и долгого нажатия.
- Основные методы: `init()`, `update()`, `getEvent()`.
- Связи: события потребляются `Navigation`, которая передаёт их текущему экрану.

**Menu / VaultMenu**
- Назначение: модели списков для главного меню (`Menu`, фиксированные 3 пункта) и для списка аккаунтов (`VaultMenu`, динамический список + пункт добавления).
- Основные методы: `init()`, `handle()`, `getSelected()`, `getCount()`, (у `VaultMenu` также `isAddAccountSelected()`, `getCurrentName()`).
- Связи: используются `MainMenuScreen` и `VaultScreen` соответственно.

# Data Model

**Account** (`src/vault/Account.h`) — структура данных одной учётной записи:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `int` | Идентификатор аккаунта в рамках текущей сессии |
| `name` | `String` | Название сервиса/аккаунта (хранится в открытом виде) |
| `login` | `String` | Логин (шифруется перед записью в память) |
| `password` | `String` | Пароль (шифруется перед записью в память) |

**Vault** (`src/vault/Vault.h`) — контейнер аккаунтов в оперативной памяти:

- Фиксированный массив `Account accounts[MAX_ACCOUNTS]`, где `MAX_ACCOUNTS = 10`.
- Счётчик `count` — текущее число аккаунтов.
- При первой настройке мастер-пароля (`vault->init()`) хранилище заполняется тремя демонстрационными записями: **GitHub**, **Google**, **Steam** (тестовые логины/пароли из кода). Это удобно для демонстрации интерфейса, но эти учётные данные при желании стоит сразу удалить/заменить после первой настройки устройства.

**Формат хранения данных** — см. раздел «Storage System» ниже.

# Storage System

**Где хранятся данные.** Данные пишутся в энергонезависимую флеш-память ESP32 через Arduino API `Preferences`, которое использует раздел NVS (Non-Volatile Storage). Используются два именованных пространства (`namespace`):

- `"auth"` — хранит хэш мастер-пароля (`pwhash`).
- `"vault"` — хранит версию схемы (`version`) и сами аккаунты.

**Формат записи одного аккаунта** (внутри namespace `"vault"`, префикс `acc<i>` — порядковый индекс при последнем сохранении):

| Ключ | Содержимое | Шифрование |
|---|---|---|
| `acc<i>_name` | Название аккаунта | Нет — хранится в открытом виде |
| `acc<i>_login` | Логин | Да — AES-256-CBC, hex-строка (`IV` + шифртекст) |
| `acc<i>_pass` | Пароль | Да — AES-256-CBC, hex-строка (`IV` + шифртекст) |

**Как происходит сохранение (`Storage::save`):**
1. Проверяется, что ключ шифрования уже установлен (`hasKey`); без разблокировки мастер-паролем сохранение невозможно.
2. Для каждого аккаунта имя пишется как есть, логин и пароль — предварительно зашифровываются `AesCipher::encryptToHex()` со случайным IV на каждое шифрование.
3. Записывается количество аккаунтов (`count`) и версия схемы хранения (`STORAGE_VERSION = 1`).

**Как происходит загрузка (`Storage::load`):**
1. Проверяется наличие ключа `version` (текущий формат) либо legacy-ключа `count` (более старый формат без версии) — при необходимости выполняется "миграция" (`migrate()`), которая на данный момент лишь проставляет актуальную версию (точка расширения на будущее).
2. Для каждого сохранённого аккаунта имя читается как есть, логин и пароль расшифровываются; если расшифровка не удалась (неверный ключ/повреждённые данные), соответствующее поле остаётся пустым, а не заполняется случайным "мусором".
3. **Важная особенность:** оригинальный `id` аккаунта не сохраняется в NVS. При загрузке каждому аккаунту присваивается новый `id`, равный его порядковому индексу в цикле загрузки (0, 1, 2, …). Таким образом, `id` актуален только в рамках текущей рабочей сессии и не является постоянным идентификатором между перезагрузками.

**Что происходит после перезагрузки устройства:**
- Если мастер-пароль ещё не задан — показывается экран установки пароля (`SETUP_ENTER` → `SETUP_CONFIRM`), после чего создаётся демо-хранилище из 3 аккаунтов и сразу сохраняется.
- Если пароль уже задан — показывается экран разблокировки (`UNLOCK`). После верного ввода из введённого пароля выводится ключ шифрования, и хранилище (`Vault`) заполняется данными, прочитанными и расшифрованными из NVS.
- Если введён неверный пароль — выводится сообщение об ошибке, поле ввода очищается, хранилище не разблокируется.

# User Interface

**MasterPasswordScreen**
- Назначение: установка мастер-пароля при первом запуске (с обязательным повторным вводом для подтверждения) либо разблокировка при последующих запусках.
- Кнопки: перебор символов (NEXT/PREV), подтверждение символа/пункта "OK" (SELECT), удаление символа (BACK).
- Переходы: при успехе — на `MainMenuScreen`.

**Main Menu**
- Назначение: выбор одного из трёх разделов — Vault, Settings, About.
- Кнопки: NEXT/PREV — перемещение по пунктам, SELECT — выбор пункта, BACK — возврат (эффективен только если есть предыдущий экран).
- Переходы: на `VaultScreen`, `SettingsScreen` или `AboutScreen`.

**Vault Screen**
- Назначение: список аккаунтов (с прокруткой, видно 4 строки одновременно) и пункт "+ Add Account".
- Кнопки: NEXT/PREV — перемещение по списку, SELECT — открыть аккаунт или экран добавления, BACK — назад в главное меню.
- Переходы: на `AccountScreen` (просмотр) или `AddAccountScreen` (добавление).

**Account Screen**
- Назначение: просмотр логина и пароля выбранного аккаунта.
- Кнопки: SELECT — показать/скрыть пароль, PREV (долгое нажатие A) — перейти к редактированию, BACK — назад в список.
- Переходы: на `EditAccountScreen` либо назад в `VaultScreen`.

**Add Account Screen / Edit Account Screen**

- Кнопки: BACK — возврат в главное меню.

# Controls

Устройство имеет две физические кнопки (`M5.BtnA`, `M5.BtnB`). Короткое и долгое нажатие (порог удержания — 800 мс) дают разные события:

| Кнопка | Действие | Событие (`ButtonEvent`) |
|---|---|---|
| A (короткое нажатие) | Выбрать / подтвердить | `SELECT` |
| A (удержание ≥ 800 мс) | Предыдущий пункт / назад по алфавиту | `PREV` |
| B (короткое нажатие) | Следующий пункт / следующий символ | `NEXT` |
| B (удержание ≥ 800 мс) | Назад / отмена / удалить символ | `BACK` |

# Serial Communication

В проекте реализован простой текстовый протокол поверх USB Serial (`SerialManager`). Команды передаются построчно (с завершением `\n`), ответы также построчны.

**Формат ответов:**
- Успех: `OK` (и, если применимо, дополнительная строка данных).
- Ошибка: `ERROR:<код>:<описание>`.

**Команды:**

| Команда | Описание | Формат запроса | Формат ответа |
|---|---|---|---|
| `LIST` | Получить список всех аккаунтов | `LIST` | `OK`, число аккаунтов, затем строки `id|name|login|password`, завершается строкой `END` |
| `GET <id>` | Получить один аккаунт по id | `GET 1` | `OK` и строка `id|name|login|password`, либо `ERROR:2:Account not found` |
| `ADD` | Добавить новый аккаунт (многошаговый ввод) | `ADD` → ответ `SEND_NAME` → отправить имя → `SEND_LOGIN` → отправить логин → `SEND_PASSWORD` → отправить пароль | `OK` с `ADDED:<id>`, либо `ERROR:3:Vault is full` |
| `DELETE <id>` | Удалить аккаунт по id | `DELETE 1` | `OK` с `DELETED:<id>`, либо `ERROR:2:Account not found` |
| `SAVE` | Принудительно сохранить хранилище в NVS | `SAVE` | `OK` с `SAVED`, либо `ERROR:4:Save failed` |

**Коды ошибок:**

| Код | Описание |
|---|---|
| 1 | Неизвестная команда |
| 2 | Аккаунт не найден |
| 3 | Хранилище переполнено |
| 4 | Ошибка сохранения |
| 5 | Отсутствует обязательный аргумент (например, id) |

Примечание: команда `ADD` через Serial работает **поверх** уже разблокированного хранилища (ключ шифрования должен быть установлен через ввод мастер-пароля на самом устройстве) — сам Serial-протокол не содержит команд аутентификации.

# Development Notes

**Как продолжать разработку:**

- **Добавить новый экран:** создать класс, наследующий `Screen` (реализовать `draw()` и `update(ButtonEvent)`), добавить его как поле в `App.h`, проинициализировать в `App::init()` и связать переходы через `Navigation::setScreen()` из экрана-инициатора.
- **Изменить логику меню/списков:** правки в `ui/Menu.*` (главное меню) или `ui/VaultMenu.*` (список аккаунтов).
- **Изменить формат хранения:** правки в `storage/Storage.cpp` (`saveAccounts`/`loadAccounts`), не забыть увеличить `STORAGE_VERSION` и добавить обработку в `migrate()`.
- **Изменить алгоритм шифрования или деривации ключа:** `security/AesCipher.*` и `security/PasswordUtil.*` соответственно.
- **Добавить новую Serial-команду:** добавить ветку в `SerialManager::handleCommand()` и соответствующий метод `cmdXxx()`.
- **Изменить набор доступных символов при вводе текста:** массив `CHARSET` дублируется в `AddAccountScreen`, `EditAccountScreen` и `MasterPasswordScreen` — при изменении редактировать во всех трёх местах (в проекте это не вынесено в общий модуль).
- **Изменить отрисовку экранов:** вся низкоуровневая графика — в `ui/Display.cpp`; сами экраны лишь передают данные для отрисовки.
